# Milestone v1.0: LCR Availability Dashboard

**Status:** ✅ SHIPPED 2026-04-07
**Phases:** 1-7
**Total Plans:** 7

## Overview

v1.0 entregou a primeira versão operacional da ferramenta de auditoria de LCRs: monitoramento por alvo com cadência configurável, validação de integridade, persistência em Postgres, alertas, dashboard de compliance, exports auditáveis e gate de readiness com smoke real contra banco.

## Phases

### Phase 1: Observability & Polling Foundation

**Goal**: Deliver the monitored data pipeline, including inventory, so every LCR/cert emits measurable availability records before we layer integrity checks.
**Depends on**: Nothing (foundation for every downstream phase).
**Plans**: 1 plan

Plans:

- [x] 01-01: Telemetry pipeline, inventory, scheduler, and coverage persistence

**Details:**
Provide a resilient poller + inventory pipeline that enforces the configured 10-minute default cadence, writes availability/coverage events into Timescale-style storage, and emits Healthchecks heartbeats so silent CRL expirations are detected before downstream logic consumes bad data.

### Phase 2: Integrity & Alerting Core

**Goal**: Validate, hash, and alert on every fetched LCR so only trusted data feeds calculations and every outage raises timely warnings before SLA breaches.
**Depends on**: Phase 1
**Plans**: 1 plan

Plans:

- [x] 02-01: Validation, cooldown-based alerting, and SLA warning helpers

**Details:**
Build the verification and alerting pipeline that admits only signed, hashed LCRs into storage and drives configurable email alerts until coverage recovers, with cooldowns and warning thresholds to avoid noisy incidents.

### Phase 3: Reporting & Compliance Governance

**Goal**: Present the data with compliance-ready dashboards, exports, and audit trails so teams can prove coverage and follow differentiated SLA insights.
**Depends on**: Phase 2
**Plans**: 1 plan

Plans:

- [x] 03-01: Reporting read models, audit UI, and CSV/PDF export builders

**Details:**
Surface everything in a reporting layer with tables, drill-downs, exports, and audit trails that prove coverage, tie dips to contracts, and support detailed evidence review by compliance and engineering.

### Phase 4: Runtime Inventory, Polling, and Persistence Wiring

**Goal**: Replace placeholders with real inventory loading, wire scheduler to validation/alerting, and move evidence flow onto persistent runtime data so the monitoring pipeline actually runs end-to-end.
**Depends on**: Phase 3
**Plans**: 1 plan

Plans:

- [x] 04-01: Database-backed inventory, runtime store, and scheduler wiring

**Details:**
Close the runtime wiring gaps identified by the milestone audit by replacing placeholder inventory loading and in-memory event handling with a persistent monitoring pipeline.

### Phase 5: Reporting E2E Wiring and Audit UX Completion

**Goal**: Connect reporting, filters, exports, and audit views to the persisted runtime data so compliance evidence is queryable and exportable from the real flow.
**Depends on**: Phase 4
**Plans**: 1 plan

Plans:

- [x] 05-01: Shared reporting query contract, functional filters, and export routes

**Details:**
Finish the reporting layer on top of the real persisted evidence by wiring dashboard filters, exports, and audit views to runtime data and closing the monitoring-to-reporting flow.

### Phase 6: Real PDF Export Delivery

**Goal**: Replace the text-based `.pdf` export placeholders with actual PDF artifact generation so compliance exports match the promised dashboard and drill-down flows.
**Depends on**: Phase 5
**Plans**: 1 plan

Plans:

- [x] 06-01: Real PDF byte generation and route delivery

**Details:**
Close the final milestone blocker by replacing fake `.pdf` downloads with real PDF bytes generated from report HTML while preserving executive and operational report variants.

### Phase 7: Production Readiness Cleanup

**Goal**: Harden the shipped system with a unified quality gate, real Postgres smoke path, retroactive validation artifacts, and minimal PDF readability cleanup.
**Depends on**: Phase 6
**Plans**: 1 plan

Plans:

- [x] 07-01: Quality gate, Postgres smoke, PDF polish, and validation backfill

**Details:**
Harden the shipped system for production readiness with a reproducible `build + typecheck + validations` gate, live Postgres smoke proof, and retroactive `*-VALIDATION.md` evidence for completed phases.

## Milestone Summary

**Decimal Phases:**

- None

**Key Decisions:**

- Decision: Use a database-backed runtime store as the source of truth for monitored targets and evidence.
- Decision: Persist alert events before delivery side effects to keep audit evidence durable.
- Decision: Keep executive and operational PDF exports as distinct report variants.
- Decision: Gate readiness with one local command plus a real Postgres smoke path.

**Issues Resolved:**

- Replaced placeholder inventory and in-memory runtime flow with persistent Postgres-backed wiring.
- Closed the broken dashboard/drill-down PDF export path with real `application/pdf` delivery.
- Added a complete reporting filter/export contract tied to the same runtime evidence path.
- Backfilled missing validation artifacts and proved runtime behavior against a live Postgres instance.

**Issues Deferred:**

- v1.1 differentiators such as pre-failure publication alerts, multi-PKI tagging, and richer SLO/burn-rate views remain for the next milestone.

**Technical Debt Incurred:**

- PDF rendering is intentionally minimal; future milestones can improve presentation without changing the export contract.
- Operational environments still need a valid `DATABASE_URL` wherever the runtime smoke or live runtime path is exercised.
