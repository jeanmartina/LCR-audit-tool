# Milestone v1.1: Multi-User Operations, I18N, and Secure Deployment

**Status:** SHIPPED 2026-04-13
**Phases:** 8-15
**Total Plans:** 8

## Overview

v1.1 turned the v1.0 monitoring core into a multi-user product. The milestone added invitation-only access, group-scoped authorization, certificate-first onboarding, predictive reporting, multilingual UI surfaces, Docker/Caddy packaging, and a real Google public-host proof over the packaged HTTPS deployment.

## Milestone Summary

**Decimal Phases:**

- None

**Key Decisions:**

- Decision: Keep access invitation-only and group-scoped from the backend outward.
- Decision: Make certificate upload, not manual CRL URL entry, the normal onboarding path.
- Decision: Package the runtime as `web + worker + postgres + caddy` with HTTPS at the edge.
- Decision: Use canonical English with translated product surfaces in `pt-BR` and `es`.
- Decision: Close v1.1 auth proof with a real Google public-host flow and defer Entra/OIDC proof to the next milestone.

**Issues Resolved:**

- Replaced SQL/manual onboarding with certificate-first flows, including packaged ZIP batch import.
- Closed reporting and export authorization gaps with group-scoped enforcement.
- Delivered Docker/Caddy packaging with real HTTPS and operator documentation.
- Completed a real public-host Google callback/session/reporting proof on the packaged deployment.

**Issues Deferred:**

- Microsoft Entra ID and generic OIDC live proof moved to the next milestone.
- Trust-list ingestion, OCSP, and richer executive analytics remain future scope.
- Operator UX polish, first-run admin bootstrap, and broader visual redesign remain backlog work.

**Technical Debt Incurred:**

- Provider verification is still manually tracked by platform admins.
- The packaged batch-import proof is validator-backed and documented, but was not exercised live in-session.
- Settings provider-save redirects still resolve to the internal origin after successful save.
