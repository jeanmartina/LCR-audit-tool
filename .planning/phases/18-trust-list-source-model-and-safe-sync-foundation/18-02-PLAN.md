# Phase 18 Plan 18-02: Trust-List Admin API, UI, Worker, and Final Validation

**Created:** 2026-04-22
**Status:** Ready for execution

## Objective

Expose the Phase 18 trust-list sync foundation to platform admins through secure API/UI controls, add Docker worker execution, and complete validation so the packaged system can register and sync ETSI trust-list sources.

## Scope

### In Scope

- Platform-admin source registration/listing API.
- Platform-admin manual sync-now API.
- Admin trust-list page using Phase 16 UI primitives and contextual hints.
- Settings navigation to the trust-list admin surface.
- Worker loop integration that runs enabled trust-list source syncs without blocking existing polling.
- i18n for `en`, `pt-BR`, and `es` for touched user-facing copy.
- Final validator coverage and full validation command pass.

### Out of Scope

- Non-admin trust-list management.
- Rich operator recovery/change-summary UX; Phase 20 owns the guided operator experience.
- Full provenance and duplicate/reimport hardening beyond what Plan 18-01 needs for initial import; Phase 19 owns this.
- Operator documentation; Phase 20 owns OPS-06.

## Threat Model

- Trust-list registration and sync-now routes must require platform-admin authorization.
- Source creation must not allow arbitrary local/metadata URL fetches; enforce HTTPS and validation in both admin and sync layers.
- UI must not imply legal/eIDAS trust validation beyond XMLDSig integrity validation.
- Worker failures must be captured as failed sync runs and must not stop existing polling forever.
- Redirects from touched POST routes must remain relative/public-origin safe.
