# Phase 18 Plan 18-01: Trust-List Persistence, Validation, and Import Core

**Created:** 2026-04-22
**Status:** Ready for execution

## Objective

Build the safe ingestion core for ETSI trust-list sources: persisted sources/snapshots/runs, XML parsing, blocking XMLDSig validation, failure-safe sync semantics, and an initial certificate import path through the existing certificate-first pipeline.

## Scope

### In Scope

- Install and lock XML parsing/signature dependencies used by the application container.
- Add runtime tables and helpers for trust-list sources, accepted snapshots, sync runs, and extracted certificates.
- Parse supported ETSI TS 119 612 LOTL/TSL XML metadata and embedded `X509Certificate` values with best-effort handling for real-world namespace variations.
- Validate XML Signature cryptographically before accepting a snapshot or importing extracted data.
- Fetch trust-list XML over HTTPS with timeouts and size limits.
- Persist every sync attempt with status, digest, sequence/territory/issue/next-update metadata, and failure reason.
- Preserve the last accepted snapshot when a later sync fails.
- Import validated extracted certificates through the existing certificate-first pipeline for the source's configured groups.
- Add a dedicated validator script for the trust-list foundation.

### Out of Scope

- Full legal/eIDAS trust-anchor policy evaluation beyond XMLDSig cryptographic validation.
- Robust unchanged-asset de-duplication and reimport change detection; Phase 19 hardens this.
- Complete provenance surfaced throughout reporting; Phase 19 hardens this.
- Operator-friendly guided trust-list onboarding screens; Plan 18-02 adds the first admin surface, Phase 20 improves UX depth.
- Documentation for operators; Phase 20 owns OPS-06.

## Threat Model

- Unsigned, malformed, or tampered XML must not mutate inventory.
- XML parsing must disable external entity/network expansion and avoid host XML tooling.
- Trust-list URL fetches must be constrained to HTTPS with timeout and response-size limits to reduce SSRF/DoS exposure.
- A failed sync must persist an auditable failed run but must never replace, delete, or hide the last valid snapshot.
- XMLDSig cryptographic validation proves document integrity for this phase; do not claim full legal trust-policy validation.
- Imported certificates must remain group-scoped and use existing authorization/import persistence rather than a parallel inventory model.
