# Phase 16 Plan: Product Visual System and Guided Form UX

**Created:** 2026-04-20
**Phase:** 16 — Product Visual System and Guided Form UX
**Requirements:** UX-01, UX-06, UX-07
**Status:** Ready for execution

## Objective

Establish the visual and form-guidance foundation for v1.2 so the product becomes easier to operate before adding trust-list and executive-dashboard complexity.

This phase must improve the existing UI in practical ways:
- create reusable UI primitives
- add clear field-level hints and examples
- improve empty/post-action guidance
- apply the system to the highest-friction current surfaces
- preserve i18n and theme support

## Scope

### In Scope

- shared UI primitives for shell, panels, fields, hints, notices, empty states, and actions
- visual cleanup of touched settings/reporting/admin surfaces
- field-level help for group defaults and provider settings
- post-save and empty states for touched flows
- i18n coverage for all new UI copy in `en`, `pt-BR`, and `es`
- validation script for UI guidance/i18n coverage

### Out of Scope

- trust-list ingestion implementation
- certificate/ZIP onboarding wizard implementation, except navigation/primitive preparation if naturally touched
- first-run admin bootstrap implementation
- executive summary dashboard implementation
- full replacement of every inline style in the app
- external component-library migration

## Implementation Plan

### Task 1 — Map Current UI Surfaces

Inspect current UI pages and identify the minimum touched surfaces for Phase 16:
- `src/app/settings/page.tsx`
- `src/app/reporting/page.tsx`
- relevant admin certificate pages under `src/app/admin/certificates/`
- current theme/global styles in `src/app/layout.tsx`
- i18n dictionaries in `src/i18n/index.ts`

Deliverable:
- concise implementation notes in the work summary or comments on what surfaces were touched and why.

### Task 2 — Add Shared UI Primitives

Create a shared UI area, preferably `src/components/ui/`, with primitives for:
- page shell/header
- panels/cards
- field wrapper with label, hint, example, and error support
- notices
- empty state
- action row / button style helpers
- metric cards if needed by touched reporting surfaces

Constraints:
- keep primitives server-component friendly
- avoid client-only dependencies unless necessary
- preserve existing CSS variable theme support
- keep API simple enough for later phases to reuse

### Task 3 — Extend i18n Dictionaries

Add canonical English keys plus `pt-BR` and `es` translations for:
- group default field hints
- provider settings guidance
- empty states
- post-save notices
- validation messages used by touched forms
- generic UI primitive labels if needed

Do not add new hardcoded user-facing strings in pages.

### Task 4 — Refactor Settings UX

Apply primitives and guidance to `src/app/settings/page.tsx`:
- user preferences section
- group defaults section
- platform settings section
- provider status/verification section

Required improvements:
- trust source / PKI / jurisdiction fields have hints and examples
- predictive window explains effect and accepted range
- provider status explains configured vs verified
- provider verification note has clear purpose
- no secrets are displayed or editable in the UI
- success/query states such as `?saved=provider` render as a proper notice

Also fix the known provider-save redirect if naturally located in this phase, because it affects post-action guidance.

### Task 5 — Add Empty/Post-Action States

Add useful states for touched surfaces:
- no groups available
- no provider configured
- settings saved
- no reporting data available if the reporting page currently renders a blank table-only state

Each state should include:
- short explanation
- one clear next action where applicable

### Task 6 — Apply Light Admin/Reporting Cleanup

Use the new primitives where low-risk on touched reporting/admin pages:
- improve headers and section hierarchy
- make zero-data states less cryptic
- avoid large rewrites unrelated to Phase 16

This task should not turn into a full redesign of all admin pages.

### Task 7 — Add UI Guidance Validation

Add `scripts/validate-ui-guidance.js` or equivalent validation checks for:
- shared UI primitive files exist
- settings page uses field guidance primitives
- required i18n keys are present in all supported locales
- no obvious hardcoded Portuguese/Spanish strings are introduced on touched pages

Wire it into `scripts/validate-all.js`.

### Task 8 — Run Verification

Run:

```bash
node scripts/validate-ui-guidance.js
node scripts/validate-i18n.js foundation
node scripts/validate-i18n.js ui
node scripts/validate-all.js
npm run typecheck
npm run build
```

If any command fails, fix the implementation or update the validator only if the validator is wrong.

## Threat Model

### Assets

- platform/provider configuration state
- group defaults and predictive settings
- user preferences
- provider verification status

### Threats and Mitigations

- **Secret exposure through provider settings UI**
  - Mitigation: UI may show configured/not configured but must not render client secrets or allow secret editing.
- **Unauthorized settings updates**
  - Mitigation: preserve existing backend authorization in settings routes; do not move enforcement to UI-only checks.
- **Misconfiguration through unclear hints**
  - Mitigation: field guidance must explain format/effect and validation must reject invalid ranges where applicable.
- **Open redirect/internal-origin redirect regressions**
  - Mitigation: post-save redirects should use the configured public origin or relative paths, not `request.url` internal container origin.
