# Phase 19 Plan 19-01: Trust-List Projection Model and Change Detection

**Created:** 2026-04-22
**Status:** Ready for execution

## Objective

Harden Phase 18 trust-list import by adding stable projection records, cross-run unchanged detection, and durable source/snapshot/run/extracted-item provenance before admin/certificate-detail surfaces consume it.

## Scope

### In Scope

- Add trust-list certificate projection persistence.
- Compute deterministic candidate keys and candidate digests.
- Record projection outcomes for imported, updated, skipped, and failed candidates.
- Skip unchanged candidates before calling `importCertificate`.
- Preserve Phase 18 XMLDSig blocking gate and failed-sync semantics.
- Add validator coverage for projection model and sync behavior.

### Out of Scope

- Rich operator recovery UI and documentation; Phase 20 owns it.
- Executive dashboards; Phase 21 owns them.
- Legal trust-policy validation beyond XMLDSig integrity validation.
- Replacing certificate-first inventory.

## Threat Model

- Unchanged trust-list candidates must not create duplicate imports or noisy updates.
- Provenance must be queryable from runtime storage, not only encoded in tags or summaries.
- Failed item projection must not invalidate an accepted signed snapshot.
- XMLDSig validation must remain before projection/import.
- Manual and ZIP imports must keep working without trust-list-specific required fields.

<task id="19-01-T3" title="Add projection change detection to sync">
  <read_first>
    - `src/trust-lists/sync.ts`
    - `src/storage/runtime-store.ts`
    - `src/inventory/certificate-admin.ts`
  </read_first>
  <action>
    Refactor `syncTrustListSource` projection loop:
    - Compute fingerprint, `candidateKey`, and `candidateDigest` for every candidate after snapshot acceptance.
    - Skip duplicate fingerprints within the current run and record projection status `skipped` with change reason `duplicate-in-run`.
    - Lookup the latest previous projection for source+fingerprint+candidateKey.
    - If previous projection has the same candidateDigest and linked certificate ID, record projection status `skipped` with change reason `unchanged` and do not call `importCertificate`.
    - If no previous projection exists, call `importCertificate` and record change reason `new-fingerprint`.
    - If previous projection exists but digest changed, call `importCertificate` and record change reason `changed-candidate`.
    - Record projection row for every imported/updated/skipped/failed outcome, linked to source, snapshot, run, extracted item, certificate, fingerprint, source path, sequence, and territory.
  </action>
  <acceptance_criteria>
    - `src/trust-lists/sync.ts` contains `candidateKey`.
    - `src/trust-lists/sync.ts` contains `candidateDigest`.
    - `src/trust-lists/sync.ts` calls `findLatestTrustListProjection` before `importCertificate` for non-duplicate candidates.
    - `src/trust-lists/sync.ts` records `unchanged` skipped projections.
    - `src/trust-lists/sync.ts` records `new-fingerprint` and `changed-candidate` change reasons.
    - `src/trust-lists/sync.ts` still calls `validateTrustListXmlSignature` before `createTrustListSnapshot` and before projection/import.

</task>
