# Phase 19 Plan 19-02: Trust-List Provenance Surfaces and Final Validation

**Created:** 2026-04-22
**Status:** Ready for execution

## Objective

Expose trust-list projection provenance in admin/reporting surfaces and complete validation so users can distinguish trust-list-derived assets from manual/ZIP imports without SQL access.

## Scope

### In Scope

- Extend admin trust-list summaries with projection counts and latest projection failure reason.
- Expose certificate-level trust-list provenance in certificate admin/detail or an equivalent admin read model.
- Add reporting read-model provenance labels/fields for trust-list-derived assets.
- Add localized copy for provenance and skipped unchanged behavior.
- Run full build/typecheck/validate-all.

### Out of Scope

- Full guided operator recovery workflows; Phase 20 owns them.
- Executive summary dashboard; Phase 21 owns it.
- Non-admin trust-list management.

## Threat Model

- UI must not imply legal trust-policy validation; keep wording to integrity/provenance.
- Reporting must not leak sources across unauthorized groups.
- Provenance display must degrade gracefully for manual/ZIP certificates.
- Skipped unchanged projections should be presented as successful no-op behavior, not hidden failures.
