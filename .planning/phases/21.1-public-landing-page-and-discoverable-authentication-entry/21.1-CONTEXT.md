# Phase 21.1: Public Landing Page and Discoverable Authentication Entry - Context

**Gathered:** 2026-04-24
**Status:** Ready for planning
**Source:** Direct user consultation during milestone-close review

<domain>
## Phase Boundary

Fix the published default URL experience so the public root path `/` no longer returns a 404 and instead serves as the product's authentication entry surface.

This phase covers only the published entry/discovery flow:
- public root landing page;
- discoverable local username/password login;
- discoverable enabled identity providers;
- authenticated-user redirect behavior from `/`.

This phase does not change invite-gated provider callback semantics, group authorization, or provider verification policy.
</domain>

<decisions>
## Locked Decisions

### Root route behavior
- The root path `/` must be session-aware.
- Unauthenticated users see a public landing page.
- Authenticated users visiting `/` are redirected directly to `/reporting`.

### Local login placement
- The local username/password form should remain on `/auth`.
- The landing page `/` should link or route users clearly into that local login flow rather than embedding the full credentials form inline.

### Identity provider presentation
- The landing page should show both enabled and disabled supported providers.
- Enabled providers are actionable.
- Disabled providers are visible but clearly unavailable.

### Landing-page depth
- The landing page should be medium-detail, not bare minimum and not marketing-heavy.
- It should include:
  - concise explanation of the product;
  - clear login entry points;
  - link to documentation if appropriate.

### Authenticated root behavior
- If a user already has a valid session and opens `/`, redirect to `/reporting`, not `/reporting/executive` and not back to the landing page.

### Delivery priority
- This is an urgent v1.2 fix discovered from real usage on the published host.
- It must land before milestone closure.

### the agent's Discretion
- Exact copy hierarchy, layout, and CTA wording.
- Whether `/auth` itself is lightly refactored to share presentation with the new landing page.
- Whether a dedicated validator is added or existing auth validators are extended, as long as root-route and auth-discovery behavior are covered.
</decisions>

<specifics>
## Specific Ideas

- Root-path users should no longer need to guess `/auth`.
- The landing should make it obvious whether sign-in is available via local credentials, Google, Entra ID, or generic OIDC.
- Disabled providers should remain visible so operators understand what exists but is not configured yet.
- The landing must work cleanly on the published host and improve the first impression of the deployed system.
</specifics>

<deferred>
## Deferred Ideas

- Invite acceptance flow redesign.
- Broader marketing-style homepage.
- Password reset UX redesign.
- Provider-proof workflow changes.
- Any non-v1.2 auth-provider expansion.
</deferred>
