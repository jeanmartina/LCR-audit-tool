# Phase 21.1 Plan 21.1-01: Public Landing Page and Discoverable Authentication Entry

**Created:** 2026-04-24
**Status:** Ready for execution

## Objective

Replace the published-root 404 with a usable public landing/auth entry flow that routes authenticated users to `/reporting`, exposes the local login path clearly, and shows supported identity providers with accurate enabled/disabled runtime state.

## Scope

### In Scope

- Add a root route at `/`.
- Redirect authenticated users from `/` to `/reporting`.
- Show a medium-detail public landing page for unauthenticated users.
- Expose a clear CTA to the existing local username/password login surface on `/auth`.
- Show supported IDPs on the landing with enabled/disabled state.
- Localize the new landing copy.
- Add validation coverage for root-route/auth discoverability behavior.

### Out of Scope

- Reworking invite acceptance flows.
- Embedding the full local login form directly on `/`.
- Changing provider callback/invite semantics.
- Marketing-site style content beyond concise product explanation and docs link.

## Threat Model

- The root route must not leak authenticated content to logged-out users.
- Provider buttons must reflect runtime configuration accurately so operators are not sent into broken flows.
- The new landing page must improve discovery without weakening invite-gated provider behavior.
- Redirect behavior from `/` must remain public-origin safe in packaged deployments.
