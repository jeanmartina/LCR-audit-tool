# Phase 21.1 Validation Strategy: Public Landing Page and Discoverable Authentication Entry

**Date:** 2026-04-24
**Status:** Ready for execution

## Validation Objective

Prove that the published root URL no longer fails with a 404, that unauthenticated users can discover how to sign in from `/`, and that authenticated users are redirected from `/` into the normal product surface.

## Required Automated Checks

Extend auth validation or add a dedicated validator to assert:

1. `src/app/page.tsx` exists.
2. The root page is session-aware and redirects authenticated users to `/reporting`.
3. The unauthenticated root renders a non-error landing surface instead of a 404.
4. The landing page exposes a clear local-login CTA that routes to `/auth`.
5. The landing page exposes supported identity providers and distinguishes enabled vs unavailable providers.
6. Provider availability is derived from runtime configuration rather than hard-coded assumptions.
7. New landing/auth copy exists in `en`, `pt-BR`, and `es`.
8. `npm run typecheck`, `npm run build`, and `node scripts/validate-all.js` pass.

## Manual/UAT Checks

- Open the published root URL `/` while logged out and confirm a landing/auth entry page appears instead of a 404.
- From `/`, navigate to local login and sign in with username/password.
- From `/`, inspect enabled and disabled provider CTAs and confirm only enabled providers are actionable.
- Open `/` while already authenticated and confirm redirect to `/reporting`.
- Confirm `/auth` still works and remains consistent with the new landing entry.

## Failure Conditions

- If `/` still returns a 404, Phase 21.1 fails.
- If local sign-in is still hidden behind an undiscoverable route, Phase 21.1 fails.
- If provider CTAs do not reflect enabled/disabled runtime state accurately, Phase 21.1 fails.
- If authenticated users land on the public entry surface instead of `/reporting`, Phase 21.1 fails.
