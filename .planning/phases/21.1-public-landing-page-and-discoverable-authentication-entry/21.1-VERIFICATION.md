# Phase 21.1 Verification: Public Landing Page and Discoverable Authentication Entry

## Verdict

PASSED - Phase 21.1 goal is achieved.

## Requirement Verification

- [x] `UX-08`: the published root URL now renders a public landing surface instead of a 404, redirects authenticated users to `/reporting`, exposes `/auth` as the local-login entry, and shows enabled/disabled identity-provider status with actionable invite-flow links for configured providers.

## Automated Checks

```bash
node scripts/validate-auth-foundation.js entry

npm run typecheck

```

## Implementation Evidence

- `src/app/page.tsx` is now the published root route and uses `getCurrentPrincipal()` plus `redirect("/reporting")` for session-aware behavior.
- `src/app/auth/page.tsx` keeps the local login form on `/auth` and adds clearer provider/invite guidance from the same published entry flow.
- `src/auth/providers.ts` adds `getExternalProviderRuntimeConfigs()` so root/auth surfaces stay aligned on non-local providers.
- `src/app/auth/accept-invite/page.tsx` now exposes stable provider anchors used by the landing/auth entry links.
- `src/i18n/index.ts` contains the new entry/auth guidance strings in `en`, `pt-BR`, and `es`.
- `scripts/validate-auth-foundation.js` now validates the new root-entry behavior and `scripts/validate-all.js` runs it by default.
- `README.md` and `docs/operators.md` document the published landing as the entry point for packaged/operator flows.

## Security And Operations Notes

- The root route remains public but does not leak authenticated content; authenticated users are redirected server-side to `/reporting`.
- Provider access remains invite-gated. The new landing only improves discoverability and does not weaken provider callback semantics.
- Disabled providers stay visible so operators can detect incomplete environment configuration before troubleshooting user access.

## Residual Manual UAT

Recommended on Docker or staging when closing the milestone:

1. Open `https://<host>/` while logged out and confirm the landing page explains local login and invite/provider access clearly.
2. Open `https://<host>/` while logged in and confirm the browser lands on `/reporting`.
3. From `/`, open `/auth` and confirm local username/password sign-in still works.
4. From `/` or `/auth`, follow a configured provider link into `/auth/accept-invite#provider-...` and confirm the expected invite-based form is focused by the anchor.
